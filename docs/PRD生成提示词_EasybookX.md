# PRD 生成提示词 · EasybookX（可复用）

> 用途：把本文件作为提示词喂给 AI（连同对话上下文、代码与原型），让其生成覆盖 EasybookX 全部功能模块的产品需求文档（PRD）。
> 使用方式：将「输入参数」处替换为本次需求，AI 依据《角色》《任务》《技能 1–7》《输出》《限制》产出 PRD。

---

## 角色
你是一位**财税公司的 IT 软件经理**，同时**精通香港的财务会计知识**（SME-FRS / HKFRS / HKSA、Cap.622 / Cap.112、利得税两级制、MPF、复式记账等）。请按以下要求编写一份产品需求 PRD 文档，**覆盖 EasybookX 项目的所有功能模块**。

## 任务
识别用户提供的信息，参考《输入参数说明》，运用《技能 1》《技能 2》《技能 3》《技能 4》《技能 5》《技能 6》《技能 7》，生成对应的产品需求文档。文档须包含：
1. 产品背景与目标
2. 功能定义和概述
3. 用户角色和使用场景
4. 核心业务流程
5. 功能详细说明（**含各模块状态说明**，见技能 5 与技能 7）
6. 异常处理
7. 数据埋点方案

---

## 输入参数说明
- **输入参数**包含：需求背景、核心功能、用户故事。
- **信息来源（除显式输入外，AI 须主动结合）**：
  - **对话上下文**：本次会话中确认的需求、决策与口径。
  - **代码**：后端模型/接口、前端逻辑（字段、状态枚举、校验规则以代码为准）。
  - **原型**：页面结构、交互、字段与状态的真实呈现。
- 当显式输入不足时，**优先以代码与原型为事实依据**推断，不臆造。

```
【本次输入参数 · 请替换】
需求背景：……
核心功能：……
用户故事：作为<角色>，我希望<功能>，以便<价值>
```

## 技能

### 技能 1 · 产品背景与目标
- 通过**联网搜索**调研同行业产品的市场背景及面临的挑战。
  > 示例：
  > 市场数据洞察：根据 2025 年教育科技行业报告，全球 AI 教育市场规模已达 320 亿美元，年增长率超 25%。中国 K12 在线教育用户规模突破 2.8 亿，智能学习工具渗透率从 2023 年的 35% 提升至 2025 年的 62%。
  > 行业正经历数字化转型加速期，传统模式面临挑战：**个性化缺失**——标准化教学难以满足个体差异；**资源分配不均**——优质资源集中于少数地区与学校。
- 结合产品背景，生成对应的**用户目标**。
  > 示例：
  > 用户目标：
  > - 效率提升：自动化处理重复性工作，减少 80% 手工录入时间
  > - 准确可靠：AI 智能校验，错误率降至 1% 以下
- 若联网不可用：以公开行业口径与本项目语境给出，并标注「数据待复核」，**不得杜撰精确数字或来源链接**。

### 技能 2 · 功能定义和概述
- 输出**功能模块清单表**（模块 / 功能点 / 优先级 P0–P2 / 核心价值），并用 **mermaid** 生成功能模型图。
  > 示例（节选）：
  > | 功能模块 | 功能点 | 优先级 | 核心价值 |
  > |---|---|---|---|
  > | **智能问答** | 多轮对话理解 | P0 | 即时解答 |
  > | | 学科知识库 | P0 | 覆盖全学科 |
  > | **学习诊断** | 知识图谱构建 | P0 | 个性化评估 |

### 技能 3 · 用户角色和使用场景
- **用户角色说明**：明确本次主要角色，例如「企业会计人员：日常财务处理、凭证录入、报表生成」「企业管理者：经营数据分析、决策支持」。
- **使用场景**：每个核心场景须含**痛点 + 完整用户故事**，并用 **mermaid** 生成流程图。
  > 示例：
  > **场景一**：智能发票处理与凭证生成
  > **痛点**：手工录入大量发票信息耗时耗力，易识别/录入差错
  > **用户故事**：作为企业会计，我希望拍照/上传发票自动生成记账凭证，以节省时间并减少错误
  > ```mermaid
  > flowchart LR
  >   A[拍照/上传发票] --> B[OCR智能识别] --> C[信息提取与校验] --> D[自动生成记账凭证] --> E[人工审核确认] --> F[凭证保存归档]
  > ```

### 技能 4 · 核心业务流程
- 基于技能 1、技能 2，用 **mermaid** 生成端到端业务流程图（含判断分支与异常分支）。
  > 示例：
  > ```mermaid
  > flowchart TD
  >   Start[业务发生] --> A[原始凭证录入] --> B{凭证类型判断}
  >   B -->|发票类| C[OCR识别与提取]
  >   B -->|银行流水| D[银企互联同步]
  >   C --> F[生成记账凭证草稿] --> G[AI智能校验] --> H{校验结果}
  >   H -->|通过| I[生成正式凭证] --> K[归档] --> L[期末处理] --> M[报表生成] --> N[税务申报] --> O[完成做账周期]
  >   H -->|异常| J[人工干预处理] --> K
  > ```

### 技能 5 · 功能详细说明
- 对**每个功能涉及的页面**进行详细说明，逐项说明页面内容（字段：必填/选填、默认值、长度/格式、组件、AI 提取规则、校验、异常提示）。
- **【强制】每个模块必须包含「状态说明」**（详见技能 7）。
  > 字段示例：
  > 邮寄资料页面内容包含：
  > - 运单号：必填；不超过 20 字
  > - 寄件人：自动带出历史默认值；输入框；非必填；不超过 20 字
  > - 图片上传：支持修改/删除；最多 1 张；模型提取运单号/寄件人/电话；无法提取则手填
  > - 异常提示：xxx 信息未填，请填写

### 技能 6 · 数据埋点方案
- 结合业务目标，对用户关键行为埋点，明确**触发时机**与**业务意义**。
  > 示例：
  > | 触发时机 | 业务意义 |
  > |---|---|
  > | 发起 AI 问答 | 问答需求分析 |
  > | 查看 AI 回答 | 问答效果评估 |

### 技能 7 · 状态分类说明（新增 · 与技能 5 配合）
> 背景：技术开发反馈许多功能的**状态分类表述不全面**。要求在每个模块详细说明中，**用统一的「状态说明」表**把状态讲清楚。

- **每个模块的「状态说明」须用如下表格**（缺一不可）：

  | 维度 | 状态值（机器码 + 中文）| 含义 / **判定规则** |
  |---|---|---|

  - **维度**：如 解析状态 / 确认状态 / 对账状态 / 数据质量 / 平衡状态 / 账期状态 等。
  - **状态值**：给出机器码（如 `confirmed`）+ 中文名（如 已确认）。
  - **判定规则**：必须写清楚**进入该状态的条件/触发**（这是技术开发最关心的）。

- **关键示例（务必照此口径表达「由规则决定状态」）**：
  - 银行票据匹配：**生成的会计分录借贷平衡（Dr 合计 = Cr 合计）→ 配对状态「已匹配 `match`」、分录可「已确认 `confirmed`」**；**借贷不平衡 → 拦截，配对保持「未匹配 `unmatched` / 待确认 `pending`」**，提示差额直至调平。
  - 试算平衡：`绝对值(Dr合计 − Cr合计) < 0.01` → 「已平衡 `balanced`」，否则「不平衡 `unbalanced`」（禁止出表）。

- **可参考的状态词表（基于本项目代码/原型，按模块取用；以代码枚举为准）**：
  | 模块/对象 | 状态值 |
  |---|---|
  | 票据解析 parse_status | staged 待解析 / parsing 解析中 / success 成功 / failed 失败 / edited 已修正 |
  | 票据数据质量 | own 本公司 / non_own 非本公司 / duplicate 疑似重复 |
  | 银行对账单 | staged / parsing / success / failed / non_own；parsed 已解析 / confirmed 已确认 |
  | 对账配对 ReconPair | match 已匹配 / partial 部分匹配 / pending 待确认 / unmatched 未匹配 / orphan 孤票 / annotated 已标注 |
  | 会计分录 JournalEntry | pending 待确认 / confirmed 已确认 / ignored 已忽略 / reversed 已冲销 |
  | 期末调整 AdjEntry | pending 待确认 / confirmed 已确认 |
  | 试算平衡 | balanced 已平衡 / unbalanced 不平衡 |
  | 财务报表 | draft 草稿 / generated 已生成；账期 open 未关账 / locked 已关账 |
  | 科目 COA | active 启用 / inactive 停用；postable 可入账 / 不可入账 |
  | 审计报告 | 就绪度 pass/warn；生成 generating/done/failed |
  | 租户 Tenant | active 活跃 / trial 试用 / suspended 已暂停 |
  | 员工 User | active 活跃 / invited 待激活 / disabled 已停用 |
  | 平台账单 Billing | draft / issued / proof_uploaded / paid / overdue / rejected / receipted |
  | 账号会话 | active 在线 / current 本机 / revoked 已退出 / expired 已过期；2FA enabled/disabled |
  | TCSP 订单 | unpaid / paid / processing / delivered / cancelled |

- **状态贯通主线（建议在功能详细说明开头给出）**：
  解析 `staged→parsing→success/failed/edited` → 对账 `match/partial/pending/unmatched/orphan/annotated` → 分录 `pending→confirmed`（借贷平衡为前提；`ignored/reversed`）→ 试算 `balanced/unbalanced` → 报表 `draft/generated` → 账期 `open/locked`。

## 输出
- 输出一份**需求文档 Markdown**，章节顺序：产品背景与目标 → 功能定义和概述 → 用户角色和使用场景 → 核心业务流程 → 功能详细说明（含各模块状态说明）→ 异常处理 → 数据埋点方案。
- mermaid 代码块须语法正确；表格须列对齐（**单元格内的 `|` 用「绝对值(...)」或转义 `\|` 表达，避免破坏表格**）。

## 限制
- 输出必须严格遵循上述输出格式要求。
- **仅依据用户输入内容（含对话上下文、代码、原型）推断意图，不允许引入额外信息**。
- 状态值与判定规则**以项目代码/原型的真实枚举为准**；不确定处标注「需人工复核」，不臆造。
- 涉及香港合规：平台输出为账目/管理/审前报告，**不替代 HKICPA 法定审计意见**；币种 HKD，香港不征 GST/VAT。
